# 06 Global-to-local 生成范式精讲

对应 `paper-reading.md`：`## 6. Global-to-local 生成范式`

## 论文原意

PhysX-Omni 不直接从图像一次性吐出整个 3D 物体，而是先做全局理解，再做局部生成。

全局阶段回答：

- 这是什么物体。
- 大概真实尺寸是多少。
- 有哪些主要部件。
- 部件之间是什么关系。
- 哪些部件会动。
- 哪些部件承担交互功能。
- 可能材料是什么。

局部阶段再逐个生成：

- 部件几何。
- 部件语义。
- 部件材料。
- 部件 affordance。
- 部件运动学属性。

## 为什么需要 global-to-local

物体物理属性不是孤立的。一个轮子是否能转，不只看它像不像圆，还看它是不是连接到底盘、轴线在哪里、转动方向是否合理。一个柜门能不能开，也取决于门板、铰链和柜体之间的关系。

全局理解给局部生成提供上下文，局部生成则让物理属性落到具体部件上。

## 代码证据

训练数据转换脚本 `dataset/2encode_representation_64_finetune.py` 会把 JSON 转成文本结构，核心字段包括：

- `Name`
- `Category`
- `Dimension`
- `Parts`
- `Group_info`

`Group_info` 中的运动类型包括：

- `A`：自由运动。
- `B`：滑动。
- `C`：绕轴旋转。
- `D`：绕点旋转。
- `CB`：旋转加滑动。
- `E`：固定到世界。

这说明局部部件不是孤立输出，而是被组织在一个全局结构里。

## 大白话说明

如果让模型只看局部，它可能知道“这是个圆”，但不知道这个圆是轮子、按钮、盘子还是装饰。  
如果让模型先理解全局，它就能知道：

- 婴儿车底部的圆更可能是轮子。
- 柜子侧面的长条更可能是门把手。
- 水桶上方的弧形更可能是提手。
- 抽屉前面的板子应该能沿滑轨平移。

## 复现检查点

评估一个输出是否真的 global-to-local，不只看 mesh：

1. `parts` 是否拆得合理。
2. 部件命名是否符合输入图。
3. `group_info` 的父子关系是否合理。
4. 可动部件的运动类型是否合理。
5. 不可见但合理存在的部件是否被过度 hallucinate。
